# DocuBot Model Card

This model card reflects on the DocuBot RAG system after implementing
retrieval and experimenting with all three operating modes.

---

## 1. System Overview

**What is DocuBot trying to do?**

DocuBot is a documentation assistant that answers developer questions
using local project files. It is designed to give grounded, accurate
answers by finding relevant documentation before generating a response,
rather than relying on general world knowledge.

**What inputs does DocuBot take?**

- A natural-language question from the developer (typed at the CLI)
- A folder of .md and .txt documentation files (the docs/ directory)
- An optional GEMINI_API_KEY environment variable for LLM-powered modes

**What outputs does DocuBot produce?**

Depending on the chosen mode, DocuBot returns either raw text snippets
from the most relevant documentation files (retrieval-only), or a
concise natural-language answer generated by Gemini that is grounded
in those same snippets (RAG). When no relevant documents are found, it
returns the explicit refusal: "I do not know based on these docs."

---

## 2. Retrieval Design

**How does your retrieval system work?**

Indexing: build_index iterates over every document, splits the text
on whitespace, lowercases each token, strips surrounding punctuation,
and records each unique word to a list of filenames in a dictionary.
Each filename is stored at most once per word to avoid double-counting.

Scoring: score_document tokenizes the query the same way and counts
how many unique query words appear anywhere in the document's lowercased
text. The score is a simple overlap count (0 if nothing matches).

Selection: retrieve uses the inverted index to find candidate documents
quickly (only docs containing at least one query word are considered),
scores each candidate, discards any with score 0, then returns the
top-k sorted by score descending.

**What tradeoffs did you make?**

The approach prioritises simplicity and speed over precision. It does
not use TF-IDF weights, stemming, or semantic embeddings, so it can
miss synonyms and return less-relevant documents when query words are
common. The upside is zero extra dependencies and near-instant index
construction on a small documentation set.

---

## 3. Use of the LLM (Gemini)

**When does DocuBot call the LLM and when does it not?**

- Naive LLM mode: Calls Gemini with the raw query and no retrieved
  context -- the model answers entirely from its training data.
- Retrieval-only mode: Never calls the LLM. Returns the raw text of
  the top-k matching documents.
- RAG mode: Calls retrieval first, then passes only the top-k snippets
  to Gemini with strict grounding instructions.

**What instructions do you give the LLM to keep it grounded?**

The RAG prompt instructs Gemini to:
1. Use only the information present in the provided snippets.
2. Not invent functions, endpoints, or configuration values.
3. Reply with exactly "I do not know based on the docs I have." when
   the snippets are insufficient.
4. Mention which files it relied on in every answer.

## 4. Experiments and Comparisons

| Query | Naive LLM | Retrieval only | RAG | Notes |
|---|---|---|---|---|
| Where is the auth token generated? | Helpful but unverifiable | Shows AUTH.md with the real answer | Concise answer citing AUTH.md | RAG most useful |
| What environment variables are required? | Helpful but may hallucinate names | Shows SETUP.md list | Clean summary citing SETUP.md | RAG cleanest |
| How do I connect to the database? | Helpful but generic | DATABASE.md snippet | Grounded answer | RAG prevents generic advice |
| Which endpoint lists all users? | Got lucky -- common REST convention | API_REFERENCE.md snippet | Cites API_REFERENCE.md | Naive LLM not trustworthy |
| Is there any mention of payment processing? | Harmful -- invented Stripe integration | Correct refusal -- nothing retrieved | Correct refusal -- "I do not know" | Best contrast between modes |
| How does a client refresh an access token? | Helpful but unverifiable | AUTH.md snippet | Helpful and grounded | RAG most trustworthy |

**What patterns did you notice?**

Naive LLM can sound very confident even when generating information
that is not in the actual codebase. For questions where the answer
happened to match common conventions it looked impressive, but it was
not grounded in the real docs. Retrieval-only was clearly better for
out-of-scope topics (like payment processing) because it correctly
returned nothing. RAG combined the strengths of both: readable answers
that are verifiably tied to the actual documentation files.

## 5. Failure Cases and Guardrails

**Describe at least two concrete failure cases you observed.**

Failure case 1 -- Naive LLM hallucination:
Question: "Is there any mention of payment processing in these docs?"
The naive LLM described a Stripe integration that does not exist in the
codebase. It should have refused to answer or said it had no information.

Failure case 2 -- Retrieval misfire on vague queries:
Question: A very short or vague query with no meaningful keywords.
The inverted index may match many documents via common words, returning
low-quality snippets ranked by coincidental word overlap rather than
actual relevance.

**When should DocuBot say "I do not know based on the docs I have"?**

1. When the inverted index returns zero candidate documents -- no query
   word appears in any doc. This is enforced by retrieve returning [].
2. When the LLM's retrieved snippets contain no information relevant
   to the question -- enforced via the RAG prompt's explicit instruction.

**What guardrails did you implement?**

- retrieve only returns documents with score > 0; if nothing scores,
  it returns an empty list.
- Both answer_retrieval_only and answer_rag check for an empty snippet
  list and return the explicit refusal string before calling the LLM.
- The RAG prompt instructs Gemini to refuse if snippets are insufficient,
  adding a second layer of protection against hallucination.

## 6. Limitations and Future Improvements

**Current limitations**

1. No semantic understanding: the keyword-overlap scorer cannot handle
   synonyms, paraphrases, or abbreviations. "DB connection" will not
   match a doc that only says "database setup."
2. No ranking within a document: each document is scored as a whole.
   Long files with a single mention of a query word score the same as
   short files entirely about that topic.
3. Common words inflate scores: stop words like "how" and "the" are not
   filtered, so vague queries can pollute relevance rankings.

**Future improvements**

1. Replace keyword overlap with TF-IDF or BM25 scoring for better
   relevance ranking without external ML models.
2. Add stop-word filtering and optional stemming to reduce noise.
3. Chunk large documents into paragraphs so retrieved snippets are
   focused passages rather than entire files.

---

## 7. Responsible Use

**Where could this system cause real-world harm if used carelessly?**

If deployed on a codebase with security-critical docs (API keys,
auth flows, database schemas), a hallucinated answer from naive LLM
mode could mislead a developer into misconfiguring authentication or
exposing sensitive endpoints. Over-trusting RAG answers without reading
the cited source files could also cause problems when documentation
itself is outdated.

**What instructions would you give real developers who want to use DocuBot safely?**

- Always read the cited source files before acting on any answer;
  treat DocuBot as a pointer to documentation, not as ground truth.
- Do not use Naive LLM mode for security-sensitive questions -- use
  RAG or retrieval-only so the answer is tied to your actual docs.
- Keep the docs/ folder current; stale documentation leads to stale
  and potentially dangerous answers.
- Treat "I do not know based on these docs." as a useful signal to
  improve documentation coverage, not as a system failure.
